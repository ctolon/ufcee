# PyO3 binding rules

## GIL discipline

- Acquire the GIL via `Python::with_gil(|py| { ... })`.
- Release the GIL around CPU- or IO-bound Rust code via `py.allow_threads(|| { ... })`.
- Never hold the GIL across `.await` or a long Rust computation.

## API surface

- Use current PyO3 idioms (`Bound<'_, T>`, `PyAny::downcast::<T>()`).
- No `&PyAny` in public function signatures.
- Functions that read a value the caller owns take `&Bound<'_, T>`; functions that consume take `Bound<'_, T>`.

## Error translation

- Define `UceeError(pyo3::exceptions::PyException)` once in `ucee_py::error`.
- Implement `From<ucee_core::Error> for PyErr` to wrap.
- Never let `unwrap()` reach the Python side; produce a `PyErr` instead.
- Map Rust error variants to Python-side subclasses (`UceeNetworkError`, `UceeConfigError`, `UceeAdapterError`).

## Module export

- The `#[pymodule]` body lists all exposed types and functions explicitly. No glob re-export.
- Type stubs (`*.pyi`) generated by `pyo3-stub-gen` and committed to `python/ucee/_native.pyi`.
- Public Python-facing names live in `python/ucee/__init__.py`; `_native` is implementation detail.

## Async bridging

- Use `pyo3-async-runtimes` (tokio variant).
- Async Rust function → Python coroutine via `future_into_py(py, async { ... })`.
- Provide both a sync facade and an async coroutine variant for high-traffic functions.

## Memory and threading

- All `#[pyclass]` types declare `Send + Sync` explicitly when they may cross threads.
- Heap allocation in hot paths is documented with a `// PERF:` comment naming the rationale.
